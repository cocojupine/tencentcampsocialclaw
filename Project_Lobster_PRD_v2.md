## 1. 破局洞察

### 1.1 真正的问题不是"年轻人不爱社交"

行业共识是 Z 世代正在经历「孤独流行病」，但数据讲述了一个更微妙的故事：

- **QQ 月活用户中，日均发送消息 ≥10 条的用户占比 78%**——他们不是不说话
- **人均加入 QQ 群数量 ≥ 5 个**——他们不是没有社交圈
- **但群聊中"有效社交转化率"（从闲聊到有结构的深度互动）不足 3%**

**结论：年轻人不是不想社交，而是启动成本太高、缺一个体面的开口理由。**

### 1.2 "语境坍塌"：群聊的结构性缺陷

传统 IM 群聊存在一个十几年未变的设计缺陷：

```
传统群聊信息流（单行道）：

[表情包] [找搭子] [八卦] [求助] [斗图] [求组队] [水群] [剧透] ...
   ↓ 全部混在一起 ↓
高价值社交意图 → 几秒内被淹没 → 转化归零
```

这叫 **"语境坍塌"（Context Collapse）**——几百人共用一条线性信息流，找搭子的、聊八卦的、讨论剧情的互相干扰，导致：

- **发出者焦虑**："我说了'缺一个'，会不会没人理我？"
- **接收者盲视**：99% 的社交信号被当作噪声过滤
- **群主无奈**：除了设公告和置顶，没有任何工具能管理群的"社交温度"

**更深层的代价是"数字语境流失"：群聊里偶尔闪现的精彩讨论——对《沙丘2》的深度解读、某个游戏的战术复盘、对番剧的共鸣——往往在几分钟内就被表情包和闲聊淹没。这些本该沉淀为"社交话题资产"的高浓度内容，没能转化为任何人际关系的催化剂。**

### 1.3 市场上现有的 AI 社交产品走错了方向

| 产品类型     | 代表                       | 做法                         | 为什么不够                                      |
| ------------ | -------------------------- | ---------------------------- | ----------------------------------------------- |
| AI 陪聊 Bot  | Character.ai、各种群聊Bot  | 替用户生成回复文本           | ❌ 越俎代庖，社交的主体变成了AI                 |
| 群聊摘要工具 | 飞书智能摘要、微信群AI助手 | 后台全量扫描聊天记录生成总结 | ❌ 隐私争议大 + Token 成本高 + 只做回顾不做催化 |
| 社交匹配 App | Soul、探探                 | 跨平台撮合陌生人             | ❌ 脱离了用户已有的社交关系链                   |

**它们的共同错误假设：AI 在社交场景中的角色是"替人说话"。**

我们的判断恰恰相反：**AI 在社交中的正确角色是不说话，而是把人说的话变成可以被看见、被响应的信号。**

## 2. 产品定义

### 2.1 一句话定位

> **群落引力场（Group Zeitgeist）：嵌入 QQ 群聊的 AI 意图信号放大器 + 腾讯数字内容分发路由 —— 不替你说话，让你的社交意图不被淹没，让腾讯内容生态自然渗透进每一次群聊对话。**

### 2.2 核心原则（不可妥协）

| 原则                           | 定义                                           | 反模式                              |
| ------------------------------ | ---------------------------------------------- | ----------------------------------- |
| **信号放大器，而非内容生成器** | 龙虾只捕捉和呈现用户的社交意图                 | ❌ 龙虾替用户写回复 / 发消息        |
| **工具调用，而非对话代劳**     | 龙虾通过 Function Calling 把能力交给用户选择   | ❌ 龙虾自动执行操作（尤其涉及支付） |
| **Opt-in 授权，而非隐性监控**  | 双轨触发均需用户授权确认，气泡仅发送者可见     | ❌ 后台全量扫描 + AI 静默发言       |
| **腾讯生态闭环，不越界**       | 所有履约能力在腾讯生态内完成                   | ❌ 对接美团等竞对平台               |
| **双轨并行，按意图强度分流**   | 复杂意图走主动配置轨道，碎片信号走被动拦截轨道 | ❌ 只有单一触发方式                 |

### 2.3 与腾讯现有 AI 能力的本质差异

| 维度         | 腾讯现有 AI 群聊能力         | 龙虾群落引力场                 |
| ------------ | ---------------------------- | ------------------------------ |
| **时间指向** | 回顾过去（总结已发生的聊天） | 创造未来（催化还没发生的关系） |
| **交互形态** | 文本回复 / 摘要段落          | 结构化 GUI 卡片（Ark/Embed）   |
| **用户角色** | 被动接受 AI 的总结           | 主动决策是否释放社交信号       |
| **核心价值** | 信息效率                     | **社交转化率**                 |

### 2.4 龙虾的生态定位：数字内容的「自然语言分发路由」

> 龙虾不仅是"群聊工具"，更是**腾讯数字内容生态的自然语言分发路由（Natural Language Distribution Router）**。

传统的内容分发依赖：搜索框 / 运营位推荐 / 好友分享——三种路径都是"人找内容"。龙虾实现的是"**内容通过社交信号自动找到人**"：

- 用户在群里聊《沙丘2》 → 龙虾捕捉 → 一张"一起看"卡片出现 → 点击直接进入腾讯视频放映厅
- 用户在群里喊"打野缺一" → 龙虾拦截 → 一张组局卡片掉落 → 点击直接拉起王者荣耀

这意味着：**腾讯视频、腾讯游戏、QQ 音乐、腾讯动漫的曝光入口，从运营位和搜索框，延展到了每一句被发出的群聊消息。**

### 2.5 S 级叙事

> **"有些群不是死了，只是缺一个开口的理由。"**

## 3. 三个核心创新点（降维打击）

市面上的 AI 社交产品都在卷"AI 怎么聊天更像人"，而这套方案的创新在于：**彻底放弃了文本生成，用 AI 去重塑群聊的"物理空间结构"。**

### 创新点 ① 架构创新：解决"语境坍塌"，首创「平行意图流」

**痛点：** 传统群聊是单行道，几百人共用一个信息流，极易导致语境坍塌。

**方案：** 龙虾的卡片机制打破了微信/QQ 十几年不变的线性信息流。它允许在主干道旁边，随时动态开辟无数个"平行子空间"。

```
改造后的群聊空间结构：

┌─────────────────────────────────────┐
│  主干道（常规聊天流）                  │
│  [表情包] [闲聊] [八卦] ...           │
├──────────┬──────────┬───────────────┤
│ 平行空间A │ 平行空间B │ 平行空间C      │
│ 🎮王者缺一  │ 🎬沙丘剧透  │ 📺一起看番   │
│ (3/5)    │ (12人参与)  │ (4/8 房间)   │
└──────────┴──────────┴───────────────┘

群聊从一个杂乱的"广场" → 变成一个个有序的"会议室"
```

**不可替代性：** 这不是发一条置顶消息——这是在群聊内部动态创建独立的、可交互的社交空间。

### 创新点 ② 交互创新：从"Chat（文本对话）"跃迁到"GUI（界面生成）"

**痛点：** 传统 Bot 只会回复文字（如"你要找游戏搭子吗？请回复1"），交互低效且枯燥。

**方案：** 龙虾不废话。它将一句干瘪的聊天记录（如"王者缺一"），**封装着色**成一张极具视觉张力的 UI 卡片。

| 传统 Bot 回复                             | 龙虾 Claw Card                                              |
| ----------------------------------------- | ----------------------------------------------------------- |
| 纯文字："检测到你想找人打王者，回复1加入" | 结构化卡片：游戏图标 + 实时坑位 + 一键按钮 + 参与者头像列表 |
| 用户需要二次理解并手动操作                | 用户一眼看懂，点击即决策                                    |

**这种"将文本瞬间转化为前端可视化组件"的能力，是下一代 AI 产品的核心交互范式——从 LLM（大语言模型）到 LIM（大界面模型）。**

### 创新点 ③ 风控创新：端云协同的"白名单嗅探"

**痛点：** 监控群聊极其耗费 Token，且有极大的隐私风险。

**方案：** 不将群聊数据全量上传大模型。采用**主动唤起 + 被动拦截双轨制 + 端云协同**的分层架构：

| 层级              | 执行位置      | 做什么                                                                 | 成本         |
| ----------------- | ------------- | ---------------------------------------------------------------------- | ------------ |
| **L1 端侧引擎**   | QQ 客户端本地 | 正则/小模型匹配关键词（"缺一""求组""剧透"）；Proactive Summon 入口监听 | ≈ 0          |
| **L2 云端 Agent** | 龙虾服务端    | 仅当用户授权后，调 LLM 生成卡片；自然语言意图解析 + 生态选项推荐       | ≈ 0.005元/次 |
| **L3 履约网关**   | 腾讯生态 API  | Function Calling 调用目标服务（跳转小程序/创建Thread/同步放映厅）      | 按需         |

**实现了隐私与成本的双重安全：原始聊天记录不上云，双轨均需用户授权确认，只有主动确认的意图才进入处理管线。**

## 4. 龙虾的四个不可替代角色

龙虾在这个闭环里绝不是一个陪聊机器人。它同时扮演四个具有技术壁垒的角色：

### 角色 ① 意图的"黄金矿工"（精准信号拦截器）

**发挥作用的地方：** 在群友海量的水群废话中，精准识别那 **1%** 真正有转化价值的"社交急迫信号"。

**典型信号模式（被动拦截轨道）：**

- 找人类：「有人吗」「缺一个」「求组队」「一起去」
- 求助类：「有人知道XX吗」「怎么弄」「急」
- 内容类：「刚看完」「剧透警告」「别剧透」

**主动唤起轨道的典型表达：**

- 活动策划类：「我们组织个观影活动吧」「周末一起打游戏吗」
- 需求征集类：「有人想买这个周边吗」「想组个XX搭子群」
- 复杂组合类：「想开个防剧透的讨论区同时一起看这部电影」

**不可替代性：** 没有龙虾，这些高价值的社交组局信号就会在几秒钟内被表情包淹没，社交机会彻底流失。

### 角色 ② 群聊空间的"折叠法师"（UI 渲染引擎）

**发挥作用的地方：** 当意图被捕捉后，动态调用前端组件。在不打断群聊主干道的前提下，在消息旁或侧边拉起独立交互空间。

**两种卡片形态：**

| 形态                   | 适用场景                   | 设计逻辑               | 视觉特征                         |
| ---------------------- | -------------------------- | ---------------------- | -------------------------------- |
| **漂浮卡 Open Card**   | 找搭子 / 约局 / 兴趣征集   | 开放性场景，最大化曝光 | 轻量无边界感，贴附原消息下方     |
| **盖楼卡 Thread Card** | 防剧透 / 已满员 / 知识沉淀 | 封闭性场景，需要边界感 | 有明确遮罩和独立入口，营造安全感 |

**不可替代性：** 它不是发一条消息——它是给你建了一个带有防剧透遮罩、带有成员统计进度的**数字房间**。这需要强大的动态 Prompt 到 UI 的渲染能力。

### 角色 ③ 跨生态的"调度员"（Function Calling 履约网关）

**最核心的价值高地：** 通过 Function Calling 连接腾讯生态内的各类服务。

| 场景           | 龙虾做的事                                      | 用户得到的体验                 |
| -------------- | ----------------------------------------------- | ------------------------------ |
| 喊"王者缺一"   | 调起王者荣耀小程序组队页                        | 一键拉起游戏，无需切出QQ       |
| 聊到《沙丘2》  | 生成"一起看"卡片 → 一键进入腾讯视频同步放映厅   | 群里的人同时看、同步聊弹幕     |
| 群里在讨论番剧 | 生成"追番集结"卡片 → 一键进入腾讯视频/B站追番页 | 同好者同频追剧、进度同步       |
| 剧透预警       | 创建 Thread 独立讨论区                          | 主群保持清爽，想聊的人有地方去 |

**关键原则：龙虾只负责调用工具并把选项摆出来，所有最终决策权在用户手中。绝不代替用户发言、绝不自动执行跨应用操作、绝不触碰任何资金流转。**

**不可替代性：** 人类完成这一套动作 = 切出 QQ → 打开其他 App → 复制链接/手动操作 → 切回 QQ。龙虾压缩成了卡片里的**一次点击**。

### 角色 ④ 腾讯数字生态的"自然语言分发路由"（Content Distribution Router）

**这是龙虾区别于所有市面 AI 社交产品的独特价值定位。**

**发挥作用的地方：** 当群聊中出现任何与腾讯数字内容相关的讨论时，龙虾自动识别并通过 Claw Card 提供最高效的转化路径——不需要用户跳出 QQ、不需要打开搜索框、不需要运营位干预。

**与腾讯生态各业务线的协同关系：**

| 群聊信号            | 龙虾识别后   | 履约结果                                |
| ------------------- | ------------ | --------------------------------------- |
| "《沙丘2》今晚看吗" | 识别影视意图 | 跳转腾讯视频「一起看」放映厅            |
| "来个打野"          | 识别游戏意图 | 跳转王者荣耀小程序组局页                |
| "这首歌太好听了"    | 识别音乐意图 | 跳转 QQ 音乐「一起听」房间              |
| "这漫画好上头"      | 识别动漫意图 | 跳转腾讯动漫/阅文小说页                 |
| "有人追这剧吗"      | 识别内容意图 | 跳转腾讯视频追剧页 + Claw Card 观影组队 |

**不可替代性：** 传统的"内容找人"依赖搜索排序和运营位推荐。龙虾实现的是**"内容通过社交信号主动找精准用户"**——每一次群聊中对某个内容的讨论，都是一次精准分发的机会。腾讯视频/游戏/音乐的 DAU 增长，从运营问题变成了一个自然发生的社交网络效应。

## 5. 核心流转机制设计：Dual-Track 交互双轨制

> 传统的 AI 社交产品只有一种交互范式：用户发指令 → AI 执行。龙虾设计了一套更符合 IM 场景的**双轨触发机制**——让用户在不同的社交意图强度下，都能以最低摩擦的方式激活龙虾的服务。

### 5.1 双轨总览

```
┌─────────────────────────────────────────────────────┐
│            龙虾 Dual-Track 交互双轨制                 │
├──────────────────────┬──────────────────────────────┤
│  轨道一：主动唤起      │  轨道二：被动拦截             │
│  Proactive Summon     │  Reactive Intercept         │
│                      │                              │
│  "我来配置一个活动"    │  "龙虾你自己看着办"           │
│                      │                              │
│  用户主动点击[🦞]入口   │  端侧引擎静默监听            │
│  配置需求 → 投放卡片    │  命中阈值 → 弹出确认气泡     │
└──────────────────────┴──────────────────────────────┘
```

---

### 5.2 轨道一：主动唤起（Proactive Summon）——"生态导购员与任务配置器"

**设计理念：** 当用户的意图较为复杂、需要组合多种能力时，提供一个"龙虾工作台"作为自然语言配置入口。

**入口位置：** QQ 群聊底部消息输入框旁边，新增专属图标 `[🦞]`

**配置面板 (Configure) 交互流程：**

```
用户点击 [🦞] 图标
         │
         ▼
┌─────────────────────────────────────────┐
│  龙虾工作台（仅自己可见）                  │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │  🦞 我想组一个《沙丘2》的观影+       │  │
│  │    防剧透讨论活动，今晚8点           │  │
│  └───────────────────────────────────┘  │
│         ↓ 自然语言输入                   │
│  ┌───────────────────────────────────┐  │
│  │  🧭 识别到你的需求，请确认配置：      │  │
│  │                                    │  │
│  │  ○ 选项 A：🎬 腾讯视频「一起看」      │  │
│  │    今晚 20:00 同步放映《沙丘2》      │  │
│  │                                    │  │
│  │  ○ 选项 B：🔒 防剧透 Thread 边栏    │  │
│  │    自动遮罩已看成员发言              │  │
│  │                                    │  │
│  │  ○ 选项 C：☑️ 两个都要              │  │
│  │    同时创建放映厅 + 防剧透边栏       │  │
│  │                                    │  │
│  │  [确认投放]  [取消]                 │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
         │
         │ 用户选择 C（两个都要）点击「确认投放」
         ▼
   一张带有「观影 + 防剧透」双重功能的
   Claw Card 掉落进群聊信息流
```

**生态导购的核心价值：**

- 龙虾不是盲目执行用户的第一个念头，而是充当**腾讯生态大管家**，主动呈现所有可用的数字履约路径
- 用户不需要知道"腾讯视频有没有一起看功能"——龙虾告诉他们
- 自然语言配置 → 结构化选项确认 → 一键投放，是最符合认知习惯的交互流

**一键投放 (Cast) 机制：**

- 用户确认配置后，面板收起
- Ark/Embed 卡片携带完整的任务参数（时间、内容、履约路径）落入群聊
- 卡片开始履约监控，倒计时、状态更新、成员管理全部自动运行

### 5.3 轨道二：被动拦截（Reactive Intercept）——"隐形僚机"

**设计理念：** 群聊中大量存在的是碎片化的、即兴的社交急迫信号，用户不会每次都打开配置面板。当一句话恰好击中高价值信号时，龙虾应该像一个隐形僚机那样，在最恰当的时刻轻轻出现。

**触发链路（S-D-A 三段式）：**

```
【SENSE — 端侧静默感知】
用户 A 在群里发："打野缺一个，来个辅助"
         │
         │ QQ 客户端本地引擎
         ▼
  正则/小模型命中关键词："缺" + "打野" + "辅助"
         │
         │ 仅上报命中信号（不上传全文）
         │ 成本: ¥0
         ▼

【DECIDE — 极简确认气泡】
仅在发送者（A）客户端弹出微小气泡：
┌────────────────────────┐
│  🦞 识别到组局意图    │
│  [投放卡片] [忽略]     │
└────────────────────────┘
         │
         │ A 点击「投放卡片」
         ▼

【ACT — 卡片自动生成并落入群聊】
Ark 卡片生成（LLM 调用 1 次）
         │
         │ 卡片出现在群聊消息流中
         ▼
  其他群友可见 + 可交互
  B 点击「上车」→ 状态同步更新
```

**被动拦截的信 号分级制度：**

| 信号强度    | 典型表达                     | 响应策略                    |
| ----------- | ---------------------------- | --------------------------- |
| 🟡 高意图   | "有人想打王者吗""周末组个队" | 弹出气泡 + 推荐多种履约路径 |
| 🟢 潜在信号 | "这剧好上头""刚看完结局"     | 仅记录，不触发，待后续确认  |

**Opt-in 机制不可绕过：** 无论信号多强，龙虾**绝不会**在没有用户确认的情况下在群里生成任何内容。气泡是发给发送者一个人的，群里其他人完全无感知。

### 5.4 双轨制的协同价值

```
主动唤起（轨道一）          被动拦截（轨道二）
    │                           │
    │ 复杂意图（多能力组合）       │ 碎片化信号（即兴急迫）
    │ 主动配置，用户主导          │ 静默感知，龙虾辅助
    │                           │
    └───────────┬───────────────┘
                │
                ▼
        同一个底层引擎
        同一套卡片形态
        同一个履约网关
                │
                ▼
        Claw Card 落入群聊
        → 状态同步
        → 履约监控
        → 腾讯生态跳转
```

两条轨道共享同一套技术架构（感知层→决策层→履约层），区别仅在于**意图的触发方式**：主动配置 vs 被动拦截。用户体验到的是——无论哪种方式，都能获得一张结构完整、可交互的 Claw Card。

### 7.2 Token 成本模型（关键！评委必问）

| 环节                        | 触发轨道 | 是否调用 LLM  | 单次成本         | 说明                               |
| --------------------------- | -------- | ------------- | ---------------- | ---------------------------------- |
| 端侧关键词匹配              | 被动拦截 | ❌ 否         | ¥0               | 本地正则/小模型，零成本            |
| 意图分类（L1/L2）           | 被动拦截 | ❌ 否         | ¥0               | 规则引擎即可覆盖                   |
| 自然语言解析 + 生态选项生成 | 主动唤起 | ✅ 是         | ¥0.002~0.003     | 仅解析用户配置需求，输出结构化选项 |
| **卡片内容生成**            | 双轨共用 | ✅ 是         | **¥0.002~0.005** | 仅当用户确认后才触发               |
| 卡片状态更新                | 双轨共用 | ❌ 否         | ¥0               | 纯数据库操作                       |
| Function Calling 解析       | 双轨共用 | ✅ 是（轻量） | ¥0.001           | 解析用户按钮点击意图               |

**核心论点：龙虾的成本优势来自"按需触发 + 双轨分流"——不是每条消息都过 LLM，被动拦截中 90% 以上仅消耗端侧引擎成本（≈ ¥0），只有用户主动确认并触发卡片生成时才调用 LLM。主动唤起轨道的 LLM 调用成本与被动拦截相当（均约 ¥0.005/次），但覆盖了更复杂的意图场景。**

对比传统方案：

| 方案                 | 每100条群消息的Token成本          |
| -------------------- | --------------------------------- |
| 全量 RAG + 摘要      | ¥0.5 ~ ¥2.0（每条都进模型）       |
| **龙虾（按需触发）** | **¥0.005 ~ ¥0.02（仅1%~2%触发）** |
| **成本降低**         | **~100倍**                        |

### 7.3 性能指标（目标）

| 指标               | 目标值  | 说明                               |
| ------------------ | ------- | ---------------------------------- |
| 端侧感知延迟       | < 50ms  | 用户发出消息到气泡弹出的时间       |
| 卡片生成延迟       | < 1.5s  | 用户点击"生成"到卡片显示的时间     |

## 8. 风控与隐私模型

### 8.1 隐私三原则

| 原则                | 定义                     | 技术实现                                     |
| ------------------- | ------------------------ | -------------------------------------------- |
| **数据最小化**      | 只处理必要的意图信号     | 端侧只上报命中的关键词+消息ID，不上报全文    |
| **用户主权 Opt-in** | 所有动作需用户显式确认   | 每次卡片生成前必须获得用户点击授权           |
| **目的限定**        | 收集的数据仅用于卡片生成 | 不用于广告推荐、不用于用户画像、不外传第三方 |

### 8.2 龙虾"四不"红线

| ❌ 不做的事         | ✅ 做的事                          |
| ------------------- | ---------------------------------- |
| 不读取私聊消息      | 仅处理群聊公开消息                 |
| 不存储聊天历史      | 只维护脱敏的结构化状态数据         |
| 不代替用户发言/行动 | 所有操作以用户名义、经用户确认     |
| 不涉及支付/金融     | 纯信息组织，交易环节由用户自行完成 |

### 8.3 可视化身份标识

龙虾生成的所有卡片和介入行为，统一带有 **"🦞 AI 助理"** 专属视觉标识：

- 卡片左上角固定龙虾 Logo
- 操作按钮旁标注"由龙虾Agent提供"
- 禁止伪装成真人或群管理员身份
